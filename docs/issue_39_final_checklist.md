## 1. ブランチ作成・テスト実行手順<br>

### 1-1. ブランチ作成<br>

> **`git checkout` ではなく `git switch` を使います。**<br>
> `switch` は Git 2.23 以降で追加された新しいコマンドで、<br>
> ブランチの切り替えに特化しており意図が明確です。<br>
> `checkout` は多機能すぎて混乱しやすいため、現在は `switch` が推奨されています。<br>

```bash
# main ブランチを最新の状態に同期させる
# なぜ: すべての実装が揃った最新の main をベースにすることで
#       チェックが不完全になるリスクを防ぐためです
git switch main
git pull origin main

# Issue #39 用の作業ブランチを作成して切り替える
# なぜ: main を汚さず、この Issue の作業だけを独立して管理するためです
# -c は --create の省略形（新規ブランチを作成して切り替える）
git switch -c feature/issue-39-final-checklist
```

```bash
# docs ディレクトリが存在しない場合は作成する

touch test/integration/final_additional_check_test.rb
```

### 1-3. 自動テスト実行（ローカル）<br>

> **テストは 1 つのコマンドで全件実行します。**<br>
> ファイルを個別に指定するよりも、フォルダ単位で一括実行する方が<br>
> 実務標準であり、テストの見落としも防げます。<br>

```bash
# Docker コンテナを起動する（既に起動中なら不要）
# -d : バックグラウンドで起動（ターミナルを占有しない）
docker compose up -d

# integration テストを全件一括実行する
# なぜ test/integration/ フォルダを指定するか:
#   このフォルダ内の全テストファイルが自動検出されて実行されるためです
#   production_final_check_test.rb と final_additional_check_test.rb の
#   両方が 1 コマンドで確認できます
docker compose exec web bin/rails test test/integration/
```

## 8. DBデータ整合性確認<br>

> **なぜ確認が必要か？**<br>
> このアプリは「論理削除（soft delete）」を使っています。<br>
> つまり習慣を「削除」しても、DBのレコードは消えずに `deleted_at` に日時が記録されます。<br>
> 機能テストだけでは「本当にDBに正しく記録されているか」は確認できません。<br>
> Rails console で直接DBの状態を確認することで、データの整合性を保証します。<br>

```bash
# ローカル環境で Rails console を起動する
docker compose exec web bin/rails console

# Render Shell の場合
# Render Dashboard → habitflow-web → Shell → bin/rails console
```

```ruby
# ① Habit の総数（論理削除されたものも含む全件数）
# なぜ: 一覧画面の件数と active 件数の差 = 論理削除件数 を確認するため
Habit.count

# ② 有効な習慣（一覧に表示される件数）
# なぜ: 画面の表示件数と一致するかを確認するため
Habit.where(deleted_at: nil).count

# ③ 論理削除された習慣（deleted_at に日時が入っているもの）
# なぜ: 削除操作が正しく DB に記録されているか確認するため
Habit.where.not(deleted_at: nil).count

# ④ 完了済みの振り返り件数
# なぜ: completed_at が正しく記録されているか確認するため
WeeklyReflection.where.not(completed_at: nil).count

# ⑤ ロック状態の振り返り件数（④と同じ件数になるはず）
# なぜ: complete! で completed_at と is_locked を同時に更新しているため
#       2つの値が一致しないと complete! が正しく動いていない
WeeklyReflection.where(is_locked: true).count
```

| # | 確認項目 | 期待結果 | 結果 |
|---|----------|----------|------|
| DB-01 | 削除した習慣の `deleted_at` が nil でないこと | 削除操作した件数分、`deleted_at` に日時が入っている | ☐ |
| DB-02 | 一覧画面の件数と `Habit.where(deleted_at: nil).count` が一致すること | 画面の表示件数 = DBの active 件数 | ☐ |
| DB-03 | 完了済み振り返りの `completed_at` に値があること | `WeeklyReflection.where.not(completed_at: nil).count` > 0 | ☐ |
| DB-04 | `completed_at` 件数と `is_locked: true` 件数が一致すること | 両方の count が同じ値 | ☐ |

## 9. N+1クエリ確認<br>

> **N+1問題とは？**<br>
> 習慣が10件あるとき、「1回の一覧取得SQL」の後に「各習慣ごとに記録を取得するSQL」が<br>
> 10回発行されてしまう問題です（1 + 10 = 11回のSQL）。<br>
> このアプリでは `index_by` や `group(:habit_id).count` で対策済みですが、<br>
> ログを見て本当に問題ないか最終確認します。<br>

```bash
# ローカル環境でログをリアルタイム表示する
# -f : 新しいログを追加で表示し続ける（Ctrl+C で停止）
docker compose logs -f web
```

> **確認手順:**<br>
> 1. 上記コマンドでログを表示したまま、ブラウザで各ページを開く<br>
> 2. ダッシュボード・習慣一覧・振り返り一覧をそれぞれ確認する<br>
> 3. 同一テーブルへの同じ条件のSQLが大量に出ていないか確認する<br>

**NG の例（N+1 が起きている）:**<br>

```sql
-- 習慣の数だけ同じ WHERE が繰り返されている = N+1
SELECT "habit_records".* FROM "habit_records" WHERE "habit_records"."habit_id" = 1
SELECT "habit_records".* FROM "habit_records" WHERE "habit_records"."habit_id" = 2
SELECT "habit_records".* FROM "habit_records" WHERE "habit_records"."habit_id" = 3
```

**OK の例（N+1 が解消されている）:**<br>

```sql
-- IN句で一括取得 = 1回のSQLで完結
SELECT habit_id, COUNT(*) FROM "habit_records"
  WHERE "habit_records"."habit_id" IN (1, 2, 3) GROUP BY habit_id
```

| # | 確認項目 | 期待結果 | 結果 |
|---|----------|----------|------|
| Q-01 | ダッシュボード表示時のSQLログ確認 | `habit_records` への同一SQLが大量に発行されない | ☐ |
| Q-02 | 習慣一覧表示時のSQLログ確認 | `GROUP BY habit_id` による一括取得になっている | ☐ |
| Q-03 | 振り返り一覧表示時のSQLログ確認 | `includes` による事前一括取得になっている | ☐ |
